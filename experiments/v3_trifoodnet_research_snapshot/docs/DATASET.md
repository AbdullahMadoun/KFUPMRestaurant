# DATASET.md — V3 export schema and how the code reads it

This is the deep-dive on what the dataset looks like on disk, what each field means, and how the loader turns it into training batches. Read this *with* `dataset_v3_adapter.py` and `dataset_integration.py` open.

## `manifest.json` — the integrity contract

```json
{
  "version": "v3",
  "export_date_utc": "2026-04-24T20:12:38.342006Z",
  "content_hash_sha8": "61ac038c",
  "source": {
    "clusters_dir": "clusters_raw_478",
    "sam_dir": "sam_outputs",
    "images_symlink_target": "/Volumes/KINGSTON/selected_5k"
  },
  "counts": { "classes": 32, "items": 4938, ... },
  "schema": { "fields": [...] }
}
```

The **`content_hash_sha8`** is the only field that matters at training time. The `V3ExportAdapter` raises if it doesn't match `master_config.yaml::data.integration.adapter.expected_hash`. This is how we keep "I trained on dataset X" verifiable across machines.

The `source` dict is provenance — tells you the export was generated by clustering 478 raw clusters from `pipeline_v2/sam_outputs/`, against images symlinked from `/Volumes/KINGSTON/selected_5k`.

## `items.jsonl` — the per-item record (4938 lines)

One JSON object per line. The full schema:

```json
{
  "sample_id":              "frame_frame_000025_00__000",   // unique key, used in episodic seeding
  "class_id":               12,                              // legacy numeric ID
  "compact_id":             11,                              // 0..31, what the model predicts
  "class_slug":             "roast_chicken",
  "class_display_name":     "roast chicken",                 // what humans (and SigLIP-text retrieval) see

  "src_image_id":           "frame_frame_000025_00",         // links back to images/<>.jpg
  "src_item_index":         0,                               // which item in the image (0,1,2...)

  "name":                   "roast chicken",                 // VLM's free-form name
  "description":            "golden-brown skin with...",     // VLM's visual description
  "vlm_label":              "roast chicken",                 // raw VLM label before canonicalization
  "vlm_canonical_label":    "roast chicken",                 // after canonicalization
  "vlm_description":        "golden-brown skin with...",     // duplicate of description (legacy)

  "bbox":                   [223.0, 118.0, 677.0, 529.0],    // [x1,y1,x2,y2] absolute pixels
  "sam_score":              0.9886,                           // SAM3's confidence
  "image_width":            1024,                              // post-resize
  "image_height":           768,
  "centrality_rank":        367,                              // ranking by how "central" this crop is to its class cluster

  "is_reference":           false,                              // true → this item is in reference/, used for few-shot library
  "image_path":             "images/frame_frame_000025_00.jpg",
  "crop_path":              "crops/frame_frame_000025_00/item_000.jpg",
  "mask_path":              "masks/frame_frame_000025_00/item_000_mask.png"
}
```

**Key field for training**: `compact_id` (0..31). That's the supervised label.
**Key fields for inspection**: `name` + `description` (the VLM's reasoning) + `centrality_rank` (high → "typical example of class", low → "weird outlier").

Items with `is_reference == true` are reserved for the Stage 3 reference library and excluded from training/eval splits.

## How the code reads this

### Layer 1: `V3ExportAdapter` (`dataset_v3_adapter.py`)

This is the bridge between the v3 export and the legacy in-memory format the rest of the code expects.

```python
adapter = V3ExportAdapter(
    export_root="/path/to/v3_2026-04-24_61ac038c",
    expected_version="v3",         # raises on mismatch
    expected_hash="61ac038c",      # raises on mismatch
    splits_path=None,              # default: writable cache dir
    train_ratio=0.80, dev_ratio=0.10, test_ratio=0.10,
    split_seed=420,
)
data = adapter.load()
```

`data.image_rows` is a list of:
```python
{
  "image_id":      "frame_frame_000025_00",
  "image_path":    "images/frame_frame_000025_00.jpg",   # relative to export_root
  "image_width":   1024,
  "image_height":  768,
  "items": [                                                # multiple items per image
    {"bbox": [...], "class_id": 12, "class_display_name": "roast chicken",
     "mask_path": "masks/.../item_000_mask.png", "crop_path": "...",
     "active": True, "excluded": False, ...},
    ...
  ],
  "review_status": "approved",   # provenance from upstream
  "use_for_export": True,
}
```

`data.split_mapping` is `{image_id: "train"|"dev"|"test"}`. Stratified — every class appears in every split (rarest classes assigned first). Cached to `splits.json`; **only recomputed if the hash or seed changes**, so multiple training runs see identical splits.

### Layer 2: `JointFoodDataset` (`dataset_integration.py::JointFoodDataset`)

Takes the adapter's `image_rows` for one split, indexes them, and serves a `__getitem__(i)` that returns a dict of:

```python
{
  "image_id":    "frame_frame_...",
  "image_tensor":   <C,H,W> resized + padded to 640
  "pil_image":      <PIL Image> resized + padded to 640
  "boxes":          <N,4> tensor of GT bboxes (in resized coords)
  "masks":          [Tensor, Tensor, ...] one per box (None if missing)
  "stage1_items":   [{"box": [...], "label": "roast chicken"}, ...]
  "stage3_query_labels":      [11, 0, ...] compact_ids of each item
  "stage3_support_episodes":  [{"support_images": [...], "support_labels": [...]}, ...]
}
```

The Stage 3 support episode is sampled from a per-class **reference library** at the same time, with **leak protection**: the support episode for an item from image X **cannot** include items from image X (the leak fix is unit-tested in `scripts/smoke_phase3.py::section_d`).

### Layer 3: `JointBatchCollator` (`dataset_integration.py::JointBatchCollator`)

Takes a list of `__getitem__` outputs, processes them through the Qwen processor (tokenizes the prompt + image), pads everything to the batch max, and returns the dict that goes into `pipeline.forward`. Handles per-stage padding (Stage 1 has variable token lengths, Stage 2 has variable item counts, Stage 3 has variable support sizes).

Each training step gets a batch like:
```python
{
  "input_ids": ..., "attention_mask": ..., "pixel_values": ...,    # Stage 1 inputs
  "images": ..., "boxes": ..., "masks": ...,                        # Stage 2 inputs
  "support_images": ..., "support_labels": ..., "query_images": ..., "query_labels": ...,  # Stage 3 inputs
  "s1_labels": ...,                                                  # Stage 1 LM target (the JSON string)
  ...
}
```

### Layer 4: `pipeline.forward(batch)` (`pipeline.py`)

Runs all three stages, computes per-stage losses, returns `{loss_total, loss_stage1, loss_stage2, loss_stage3}`. The `train_joint.py` loop sums weighted losses, backprops, applies per-stage learning rates.

## How dataset construction relates to `pipeline_v2/`

The v3 export was **generated by `pipeline_v2/`** (in the parent repo, sibling to this experiments dir):

```
selected_5k/  (3702 raw images)
   │
   ├─► Stage 1: Qwen2.5-VL-72B-AWQ → vlm_outputs/<image>.json
   │     (free-form {name, description, bbox} per detected item)
   │
   ├─► Stage 2: SAM3 + bbox prompts → sam_outputs/<image>/
   │     (segmentation masks + masked crops)
   │
   └─► Stage 3: DINOv2 + SigLIP → cluster → 478 raw clusters
         (then manually merged to 32 final classes by Madoun)

→ exports/v3_2026-04-24_61ac038c/    ← what this branch reads
```

So the `class_display_name` you see in `items.jsonl` is the **manually-merged label from Madoun's clustering review**, not raw VLM output. The `name` and `description` fields preserve the original VLM output for traceability.

If you want to re-create the dataset (e.g., with new images), see `pipeline_v2/pipeline.py`.

## Splits — important detail

The 80/10/10 split is **image-level stratified** (not item-level), seed = 420. It's recomputed deterministically from `image_id` only, then cached. As long as you don't change:

- the v3 export's `content_hash_sha8`
- `split_seed` in `master_config.yaml`
- `train_ratio` / `dev_ratio` / `test_ratio`

…you'll get the same dev images on every machine, every run. **Every run we've reported numbers on uses dev split = 370 images / 514 items.**

## Reference library (for Stage 3 few-shot)

The `reference/` folder contains 32 sub-folders (one per class), each with ~5 hand-picked typical examples. These images are also in `images/` and `items.jsonl` (with `is_reference: true`), but the reference folder makes them easy to grab.

Stage 3 at training time samples 5-way 5-shot episodes:
- 5 classes drawn from the 32
- 5 support images per class (drawn from items where `is_reference==true` OR from the train split's items of that class)
- 1 query image per class (the actual current sample being classified)
- The classifier sees support → predicts query

Stage 3 at inference time uses a fixed reference index built from `reference/`.

This is also where the `vector-db-classifier` branch differs: instead of running an episodic transformer over support+query, it embeds each `reference/<class>/*.jpg` once into a per-class prototype, then classifies a query crop by cosine similarity to those prototypes. Same data, simpler classifier.
